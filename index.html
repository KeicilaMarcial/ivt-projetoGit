Conflito


Um conflito de versões ocorre quando duas ou mais versões commitadas de um código alteraram a mesma linha de um arquivo. Nesse caso, o git não reconhece qual altereção deve ser seguida. Assim o alerta de conflito é gerado.

Na imagem abaixo houve conflito em várias linhas do código, pois houve duas alterações simultâneas commitadas por pessoas diferentes nessas linhas.

O código envolvido com o comentário HEAD corresponde ao código local, enquanto o restante envolvido por ======= corresponde ao código remoto.

Tags


As tags são usadas para marcar pontos importantes no histórico do projeto.

O comando git tag [-comandos (opcional)] "nome-da-tag" [hash do commit (opcional)] possibilita a criação de tags em commits específicos.

Stash


O comando git stash "esconde" as alterações do código que não foram commitadas. As alterações são salvas em outro arquivo, sejam elas staged ou unstaged.

É possível exibir os stashs realizados com git stash list ou recuperá-los com git stash pop.

Merge x Rebase


Os comandos git merge e git rebase fazem a mesma coisa de maneiras diferentes.

Ao usar o Merge, as alterações de outra branch serão misturadas com as alterações da sua branch atual. Esse comando mantém a distinção de histórico entre as branchs.

Ao usar o Rebase, as alterações da sua branch atual serão adicionadas ao final do histórico da outra branch como commits adicionais. O código da sua branch atual receberá as alterações da outra branch. Esse comando não mantém a distinção de histórico entre as branchs.

Repositório Local


Caso 1: repositório do zero


  • Crie um direório para conter o projeto
  • Navegue até esse diretório
  • Digite: git init
  • Digite git add para adiconar os aquivos
  • Digite git commit

    • Caso 2: Um novo repositório de um projeto existente


  • Vá para o diretório que contém o projeto
  • Digite: git init
  • Digite git add para adiconar os aquivos
  • Provavelmente, você desejará criar um arquivo .gitignore imediatamente, para indicar todos os arquivos que não deseja rastrear. Use git add .gitignore também.
  • Digite git commit

    • BONUS: Conectando ao github


  • Vá para Github
  • Log na sua conta
  • Clique no botão novo repositório no canto superior direito. Você terá a opção de inicializar o repositório com um arquivo README
  • Clique no botão “Criar repositório”.

    • Agora siga o segundo conjunto de instruções “Envie um repositório existente ...”


    $ git remote add origin git@github.com:username/new_repo $ git push -u origin master

    Na verdade, a primeira linha das instruções dirá


    $ git remote add origin https://github.com/username/new_repo

    Porem utilizando git@github.com: username / new_repo
    em vez de https://github.com/username/new_repo, não será
    necessário digitar sua senha toda vez que enviar algo para o github

    Exclusão de Branches


    Caso 1: Excluir Branch Local


    $git branch -d nome_do_branch $git branch -D nome_do_branch

    Diferenças entre -d e -D


  • -d é um atalho para --delete exclui o branch localmente apenas se você já tiver feito push ou merge com seu branch remoto.
  • -D é um atalho para--delete --force , ou seja, exclui o branch independente de seu status de push ou merge

    • Caso 2 : Excluir Branch Remoto no Git


    git push origin :nome_do_branch_remoto
  • Ps:é brigatório o uso do : antes do nome do branch.

    • Comando Alternativo


    git push origin --delete nome_do_branch_remoto
  • PS: Para o caso de receber o erro error:unable to delete ‘nome_do_branch_remoto’: remote ref does not exist error: failed to push some refs to ‘git@repositorio’ ,é porquê alguém excluiu o branch antes de você.

    • Mover Commits


    Caso 1: Mover Para uma Branch Existente


    git checkout existingbranch git merge master # Bring the commits here git checkout master git reset --keep HEAD~3 # Move master back by 3 commits. git checkout existingbranch
  • --keep : preserva quaisquer alterações não confirmadas que você possa ter em arquivos não relacionados ou anula se essas alterações precisarem ser substituídas - da mesma forma que git checkoutocorre.

    • Caso 2: Movendo para uma nova branch


    git branch newbranch # Create a new branch, containing all current commits git reset --keep HEAD~3 # Move master back by 3 commits git checkout newbranch # Go to the new branch that still has the desired commits
  • Este método funciona criando uma nova ramificação com o primeiro comando ( git branch newbranch), mas sem mudar para ele. Em seguida, revertemos a ramificação atual (principal) e mudamos para a nova ramificação para continuar trabalhando.
